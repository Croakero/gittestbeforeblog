Make

You need to write down all the make commands on some files(called Makefile) and then every time you wanna build the files you should run make,which would run all the necessary commands for you so you donot need to remember them yourself.

Everything on the left of the colon is the target and everything on the right is the dependencies and all lines below are the sequence of operations known as the rules for once you have the dependencies how you build these targets.

You must use tab to indent the rules if you not,make would not work.

Version

What are the rules for publishing the versions and how to change version numbers?

Some are just dictated by time,people have strong opinions on exactly the scheme should be and you cannot predict which people would use.

Semantic numbering is a solution,which gives each of the number separated bydots in a version number a particular meaning and we gave rules to increment these numbers.

major.minor.patch Ususally security fixes would increase the patch number.

If you add something in your labraries you increment the minor version.

Major version was incremented if you make a backward imconpatible change.

Lock Files

Lock files are to make sure that you don't accidentally update something.

The lock file at its core is a list of dependencies and which version of them you're currently working.

You might want your build to be fast,it you use lockfile,it will just use lockfiles to get previously built dependencies. Another reason to use lockfiles is to build reproducible software,allowing you to freeze the ecosystem as of this version that I have checked.

Continuous Integration

Continuous integration is an event triggered system which let you decide what happens when actions that you defined happens. It allows you to tie something related to your code.

You add a file in your repository and the file specifiles dependency cycles again sort of what we saw in makefiles.

Testing

•   test suite:It is just a collection of tests that are usually run as a unit.The most known of it is a unit test which tests a very particular minor thing of the whole project.

•   Integration tests tests different subsystems of a program.

•   Regression tests tests things that are broken in the past.

•   Mocking is the idea of being able to replace parts of your system with a sort of dummy version of itself that behaves in a way tht you control.
